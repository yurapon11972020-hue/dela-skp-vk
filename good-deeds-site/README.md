# Сайт добрых дел: `/dela/` и `/admin/`

Готовый статический проект для **GitHub Pages** с базой и авторизацией через **Supabase**.

## Что умеет сайт

### Публичная часть `/dela/`
- пользователь пишет, какое доброе дело он сделал;
- указывает, **от кого** это доброе дело;
- отправляет запись на модерацию;
- видит общую ленту уже одобренных дел;
- может ставить лайки.

### Админка `/admin/`
- вход по **email + password** через Supabase Auth;
- просто открыть страницу `/admin/` недостаточно;
- система проверяет, есть ли у вошедшего пользователя права администратора;
- админ может:
  - одобрять записи;
  - отклонять записи;
  - редактировать текст и автора;
  - накручивать лайки;
  - закреплять запись;
  - менять порядок закрепления;
  - удалять записи;
  - писать заметку администратора.

---

## Важный момент по безопасности

**На одном только GitHub Pages нельзя безопасно хранить пароль от админки внутри кода сайта.**

Поэтому здесь сделана правильная схема:
- фронтенд лежит на GitHub Pages;
- вход в админку идёт через **Supabase Auth**;
- право администратора хранится в таблице `admin_users`;
- пароль не лежит в репозитории и не вшит в JavaScript.

То есть:
- пользователь **может попробовать открыть** `/admin/`;
- но без входа он ничего не увидит;
- без записи в `admin_users` он не получит права администратора;
- через консоль браузера нельзя просто «включить себе админку», потому что доступ контролируется политиками базы.

## Шаг 1. Создайте проект в Supabase

1. Зарегистрируйтесь в Supabase.
2. Создайте новый проект.
3. Откройте **SQL Editor**.
4. Вставьте содержимое файла `supabase.sql`.
5. Выполните SQL.

Это создаст:
- таблицу `good_deeds`;
- таблицу `post_likes`;
- таблицу `admin_users`;
- функции для отправки заявок и лайков;
- RLS-политики безопасности.

---

## Шаг 2. Создайте администратора

1. В Supabase откройте **Authentication → Users**.
2. Создайте пользователя с email и password.

Например:
- email: `admin@example.com`
- password: ваш сложный пароль

После этого нужно выдать этому пользователю права администратора.

Откройте **SQL Editor** и выполните:

```sql
insert into public.admin_users (user_id)
select id
from auth.users
where email = 'admin@example.com'
on conflict do nothing;
```

Замените `admin@example.com` на ваш email администратора.

Теперь этот пользователь сможет входить в `/admin/`.

---

## Шаг 3. Подставьте настройки в `config.js`

Откройте в Supabase:
- **Project Settings → API**

Скопируйте:
- `Project URL`
- `anon public key`

Откройте файл `config.js` и замените значения:

```js
export const SUPABASE_URL = 'https://YOUR_PROJECT.supabase.co';
export const SUPABASE_ANON_KEY = 'YOUR_PUBLIC_ANON_KEY';
```

> `anon key` можно хранить на фронтенде. Это нормально. Безопасность здесь держится не на сокрытии ключа, а на **RLS-политиках** и проверке администратора.

## Шаг 4. Проверьте локально

Например, если у вас установлен VS Code:

Или через Python:

## Шаг 5. Залейте на GitHub

1. Создайте новый репозиторий на GitHub.
2. Загрузите все файлы из этой папки.
3. Закоммитьте изменения.
4. Откройте **Settings → Pages**.
5. Включите GitHub Pages для ветки `main`.

После публикации у вас будут адреса:
- `https://USERNAME.github.io/REPO/dela/`
- `https://USERNAME.github.io/REPO/admin/`

## Как это работает технически

### Отправка доброго дела
Публичный сайт не вставляет запись напрямую в таблицу.
Он вызывает функцию:
- `submit_good_deed(...)`

Эта функция всегда создаёт запись со статусом:
- `pending`

То есть обычный пользователь не может через фронтенд сразу опубликовать запись как `approved`.

### Лайки
Публичный сайт вызывает функцию:
- `increment_good_deed_like(...)`

Для лайка используется токен браузера из `localStorage`.
Это защита от повторного лайка с одного и того же браузера.
Полностью остановить накрутку на открытом публичном сайте нельзя, но это уже лучше, чем просто открытый `update likes_count`.

### Админка
Админка входит через Supabase Auth.
После входа вызывается функция:
- `is_admin()`

Если пользователя нет в таблице `admin_users`, доступ к панели не даётся.

## Что именно защищено

### Защищено
- пароль администратора не лежит в коде сайта;
- обычный посетитель не может получить права администратора только через DevTools;
- обычный посетитель не может напрямую редактировать заявки через публичную страницу;
- обычный посетитель не может одобрять заявки;
- обычный посетитель не может закреплять записи;
- обычный посетитель не может удалять записи.

### Не защищено на 100%
- путь `/admin/` можно увидеть и открыть, если знать адрес;
- лайки с публичного сайта теоретически можно накручивать через новые браузерные токены.

Если нужна более жёсткая защита лайков, можно позже добавить:
- авторизацию пользователей;
- captcha;
- rate limiting через Edge Functions;
- модерацию по IP или device fingerprint.

## Быстрый чек-лист

1. Выполнить `supabase.sql`.
2. Создать admin-пользователя в Supabase Auth.
3. Добавить его в `public.admin_users`.
4. Заполнить `config.js`.
5. Залить проект в GitHub.
6. Включить GitHub Pages.
7. Проверить:
   - отправку заявки через `/dela/`;
   - вход через `/admin/`;
   - одобрение и отклонение заявок;
   - лайки;
   - закрепление записи.

---

## Если хотите совсем скрыть `/admin/`

Полностью скрыть адрес админки в интернете нельзя.
Но можно сделать неудобным его угадывание:
- переименовать папку `admin` во что-то нестандартное;
- убрать ссылки на админку с публичной части;
- повесить собственный домен и отдельный поддомен.

Но **главная защита должна быть не в скрытии адреса**, а в нормальной авторизации и RLS — что здесь уже сделано.
